# Introduction

A Lox parser must be defined in a dedicated Go package (the name is not
important). It must contain at least one lox grammar file (with a `.lox`
extension) that includes at least one lexer and one parser section. Additionaly,
the package must contain at least one Go source file (`.go`) defining the
`Token` type and a parser type. The parser type must be a `struct` that embeds
the type generated by the `lox` tool.

The `lox` tool processes both the grammar and the Go source files in the package
to generate the actual parser and lexer. It will examine your parser type for
methods corresponding to parser productions and verify that the return types of
actions align with the term types in the grammar. Certain parser features are
enabled only if your Go source implements the required methods.

The easiest way to get a feeling for Lox is to look at an example. The following
parser will parse a list of comma-separated integers inside parenthesis (e.g.
`(1, 217, -77)`). You can copy the files to an empty Go module, run `lox .`,
then run `go run .`.


```lox
// === parser.lox ===

@lexer // Marks the beginning of the lexer section, where tokens are defined.

NUMBER = [1-9][0-9]*

OPAREN = '('
CPAREN = ')'
COMMA  = ','
MINUS  = '-'

// Discard empty space, including new lines and tabs.
@frag [ \r\n\t]+  @discard

@parser // Marks the beginning of the parser section, where parser rules are defined.

// number_list is the root parsing rule as indicated by the @start keyword. It
// is composed of a single production. Notice that token terms that are simple
// literals can be referenced by the literal (e.g. ',') instead of the token
// name (but the token still needs to be declared in the lexer section). The
// @list is a convenience term modifier that will expect a comma-separated list
// of `number`. The '?' cardinality indicates that the list is optional (so that
// `()` is a valid input).
@start number_list = '(' @list(number, ',')? ')'

// number has two productions. 
number = NUMBER         // Positive (unsigned) numbers
       | '-' NUMBER     // Negative numbers.

```
```go
// === parser.go ===

package main

import (
  "fmt"
  gotoken "go/token"
  "log"
  "strconv"

  "github.com/dcaiafa/loxlex/simplelexer"
)

func main() {
  input := "(1, 217, -77)" // Feel free to try different inputs.

  // While Lox generates the lexer state machine, you are responsible for
  // providing the actual lexer implementation. In most cases, `simplelexer` is
  // sufficient. If you prefer not to have external dependencies, you can
  // implement your own lexer or copy `simplelexer` locally.
  lex := simplelexer.New(simplelexer.Config{
    StateMachine: new(_LexerStateMachine),
    File:         gotoken.NewFileSet().AddFile("input", -1, len(input)),
    Input:        []byte(input),
  })

  var parser myParser

  // parse is a method implemented in the embedded `lox` type.
  ok := parser.parse(lex)
  if !ok {
    // By default, the generated parser indicates whether it succeeded in
    // parsing the input or not. For production use, you will likely want to
    // implement more robust error handling. See "Error Handling and Recovery"
    // for more details.
    log.Fatal("failed to parse")
  }

  fmt.Println(parser.result)
}

// You must define a type named `Token` because the `lox` tool expects your
// lexer to produce tokens of this type. Since we are using `simplelexer` as the
// lexer implementation, we need to alias its `Token` type to match the expected
// type.
type Token = simplelexer.Token

// You must define a parser type. The name of this type is not important, but it
// must embed the generated `lox` type. The `lox` tool will match productions to
// methods of your type.
type myParser struct {
  // Identifies this type as THE parser. Also provides parser state and
  // primitives.
  lox

  // You can have anything else you want in your parser. In this case we need a
  // field to store the list of parsed numbers.
  result []int
}

// The "on_" prefix indicates to `lox` that this method is the action associated
// with a production rule named after the "on_" prefix—in this case,
// "number_list".
//
// The method parameters must correspond to the terms of the production. If a
// term's value is not needed, its corresponding parameter can be named "_" to
// indicate it is ignored, but the parameter must still be present in the method
// signature. The type of each parameter must match the term's value type. For
// token terms, the type is always `Token`, while for rule (non-terminal) terms,
// the type is determined by the return type of the referenced rule's action 
// method.
//
// In this example, both `number` actions return `int`, but due to the special
// cardinality modifier `@list`, the method receives a slice (`[]int`)
// representing a list of the encompassed type.
func (p *myParser) on_number_list(_ Token, nums []int, _ Token) any {
  p.result = nums
  return nil
}

// The "number" rule has multiple productions, each requiring a separate action
// method. Since `lox` relies on the rule name following the `on_` prefix and
// the method parameters to match an action to a production, and Go does not
// support method overloading, you must provide a unique suffix to differentiate
// each action method. The suffix can be any valid character sequence following
// a double underscore ("__").
//
// The following action method corresponds to the production:
//
//  number = NUMBER
//
// In this example, the suffix "__positive" is arbitrary and serves to
// distinguish this method from other actions for the "number" rule.
func (p *myParser) on_number__positive(n Token) int {
  v, err := strconv.Atoi(string(n.Str))
  if err != nil {
    panic(err) // for illustration purposes; you can do better.
  }
  return v
}

// This action method matches the production:
//
//  number = '-' NUMBER
func (p *myParser) on_number__negative(_ Token, n Token) int {
  v, err := strconv.Atoi(string(n.Str))
  if err != nil {
    panic(err)
  }
  return -v
}
```
